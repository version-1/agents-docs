# Code Review スキル

このドキュメントは、コードレビューを行うときの
**思考の枠組み・判断基準・進め方** を定義します。

これは実装方法のガイドではありません。
「どう直すか」ではなく **「何を疑い、何を優先して指摘するか」** を示します。

---

## このスキルを使う場面

- 実装済みの差分をレビューするとき
- 仕様違反、回帰、危険な設計変更、検証不足を見つけたいとき
- レビュー結果を重要度順で整理し、次の判断材料を作りたいとき

## このスキルを使わない場面

- 実装方針やリファクタリング計画を作りたいとき
- 実装そのものを行うとき
- 言語・フレームワーク固有の書き方だけを確認したいとき

上記の場合は、`code-refactor`、`coding`、`languages/` 配下の対応スキルを優先する。

---

## 目的

- 実装された変更が仕様・制約を満たしていることを確認する
- 変更のリスクを早期に可視化し、必要な是正や検証を促す
- レビュー品質のばらつきを減らし、判断基準を揃える

## このスキルが提供するもの

- レビュー時の観点と優先順位
- 重大度の付け方
- `findings first` の出力形式
- 不明点や追加検証の整理方法

## このスキルが提供しないもの

- パッチや具体的なコード変更
- 実装者向けの詳細な設計案
- 承認 / 却下の最終意思決定

---

## 絶対ルール

- 事実と推測を混同しない
- 差分を起点にレビューする
- 重大度の高い指摘を先に出す
- 根拠、影響、問題になる条件を説明できる指摘だけを出す
- 不確実なものは断定せず、質問や懸念として分ける

---

## レビュー対象の境界

- 基本は **差分中心** で見る
- 差分の影響で壊れうる関連箇所は読む
- 未変更ファイルでも、差分が依存している契約・呼び出し先・テストは確認対象に含めてよい
- 今回の差分と無関係な既存不具合は、レビューの主題にせず補足扱いにとどめる

---

## レビューの進め方

レビューは次の順で行う。

1. 差分を読み、何が変わったかを把握する
2. 仕様・公開契約・データ互換性を壊していないかを見る
3. セキュリティ、整合性、並行性、破壊的変更など High リスクを先に疑う
4. エラー処理、監視、テスト、回帰防止の不足を確認する
5. 将来変更耐性、可読性、スタイルなどの二次観点を見る
6. 指摘を重要度順に整理し、不明点と追加検証を分けて出す

軽微な命名やスタイルより、仕様違反・回帰・高リスク変更・未検証の方を優先する。

---

## 指摘として成立する条件

指摘を出すときは、最低限次を含める。

- 何が問題か
- どこで問題になるか
- なぜ問題だと判断したか
- どの条件で壊れるか、または何が未検証か

指摘に含めなくてよいもの:
- 具体的な修正コード
- 実装者の意図を決め打ちした断定

---

## 重要度の付け方

### High

次のいずれかに当てはまるもの。

- 仕様違反、公開契約破壊、既存挙動の破壊
- セキュリティ、認可、情報漏えい
- データ破壊、整合性破壊、移行失敗
- 並行性、排他、冪等性の欠陥
- 本番障害や重大回帰につながる可能性が高いもの

### Medium

次のいずれかに当てはまるもの。

- 重要なケースが未検証で、回帰を見逃しやすい
- エラー処理、監視、境界条件への考慮が不足している
- 将来の変更コストや障害調査コストを明確に増やす

### Low

次のいずれかに当てはまるもの。

- 命名、可読性、軽微な構造改善
- 直ちに障害にはつながらないが、理解コストや保守コストを上げるもの

補足:
- カテゴリ名だけで重大度を決めない
- たとえばテスト不足でも、公開契約変更の未検証なら High になりうる
- 命名でも意味誤解により挙動を読み違えるなら Medium 以上になりうる

---

## レビュー観点

### 1. 仕様・契約・互換性

- 仕様変更が明示されているか
- 公開 API や既存レスポンスの互換性が壊れていないか
- スキーマ変更が既存データを破壊しないか
- 既存クライアントやバッチの暗黙依存を壊していないか

### 2. セキュリティ・権限

- リソース単位の認可が抜けていないか
- マルチテナント / ユーザー分離が守られているか
- 外部入力を無検証で使っていないか
- ログやレスポンスに機密情報が漏れていないか
- CSRF / CORS / Cookie 属性の前提が崩れていないか

### 3. データ整合性・並行性

- 一意制約、外部キー、チェック制約などの前提が守られているか
- アプリ側バリデーションのみに依存していないか
- 部分成功時の整合性が考慮されているか
- 冪等性が必要な操作で二重実行が起きないか
- 競合更新や read-modify-write が安全か

### 4. エラー処理・異常系・観測可能性

- エラーを握りつぶしていないか
- 想定内 / 想定外エラーが区別されているか
- タイムアウト、キャンセル、外部依存失敗時の挙動が明確か
- ログ、メトリクス、トレースが必要十分か
- 失敗時に原因追跡できる情報が残るか

### 5. アーキテクチャ・責務

- レイヤー境界を越えていないか
- Domain / UseCase に I/O 詳細が入り込んでいないか
- 1つの関数やクラスが責務を抱えすぎていないか
- 同じビジネスルールが複数箇所に散っていないか
- public API や export が不要に増えていないか

### 6. テスト・検証

- 仕様上重要な振る舞いがテストで観測できているか
- バグ修正なら再現テストや回帰テストがあるか
- 破壊的変更に検証手順があるか
- テストが実装詳細に依存しすぎていないか
- 層ごとの役割分担が適切か

### 7. 将来変更耐性・性能

- 分岐増殖だけで新要件に対応する構造になっていないか
- 特定実装への結合が強すぎないか
- “賢すぎる” 実装で理解コストが上がっていないか
- ループ内 I/O、N+1、全件取得後フィルタなどの性能問題がないか
- リソースリークや不要な保持がないか

### 8. スタイル・命名

- typo やスタイル違反がないか
- 命名が意味を正しく伝えているか
- コメントが正確で最新か
- 例外的な仕様や特別な理由が明示されているか

---

## 出力形式

出力は次の順で行う。

1. **指摘一覧**
   - 重要度 High / Medium / Low
   - ファイル / 行
   - 問題の要約
   - 根拠と影響
   - 壊れる条件、または不足している検証
2. **不明点・仕様確認が必要な点**
3. **テスト / 検証の不足**
4. **変更点の短い要約**

必要な指摘がない場合:
- `重大な指摘はなし` と明示する
- そのうえで残るリスクやテスト不足があれば書く

原則:
- 良い点の列挙を主目的にしない
- `悪い点` のような感想セクションは作らない
- 点数評価は行わない

---

## レビュー時の注意

- 実装者の意図を好意的に補完しすぎない
- 一方で、根拠のない断定もしない
- 指摘が出ない場合でも、見た範囲と残る不確実性を意識する
- 軽微なスタイル論争で、重大なリスクを埋もれさせない
