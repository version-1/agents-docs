# レビュー観点

## アーキテクチャ・責務

### レイヤー境界
- 依存方向が正しいか（内側 → 外側への参照が混ざっていないか）
- Domain にフレームワーク / DB / HTTP など I/O 依存が入り込んでいないか
- UseCase が UI / DB の詳細（SQL、HTTP ヘッダ、ORM 構造）を直接扱っていないか
- Infrastructure がビジネスルールを持っていないか（インフラ都合の分岐が増えていないか）

### 責務の混在・凝集度
- 1つの関数 / クラスが「取得・変換・検証・保存・通知」を同時に担っていないか
- “便利関数” が複数レイヤーから呼ばれ、責務が膨張していないか
- UI 都合の整形や文字列生成が Domain / UseCase に混ざっていないか
- 同種のビジネスルールが複数レイヤーに散在していないか

### モジュール境界・公開 API
- public API が増えすぎていないか（外部に漏れる責務が増えていないか）
- 内部実装が export / public になっていないか
- 循環依存や「とりあえず import」が発生していないか


## データ・整合性

### 破壊的変更・互換性
- スキーマ変更が既存データを破壊しないか（drop / rename / 型変更 / NOT NULL 化）
- 既存 API のレスポンス互換性が壊れていないか（フィールド削除・意味変更）
- 既存のクライアントやバッチが暗黙に依存している挙動を壊していないか

### 整合性・制約
- 一意制約・外部キー・チェック制約など、守るべき制約が明示されているか
- アプリ側バリデーションのみに依存していないか（並行実行で破れる可能性）
- 部分成功（片方だけ成功）時の整合性が考慮されているか

### 冪等性
- リトライや重複リクエストで二重登録・二重課金・二重送信が起きないか
- 同じ入力に対して同じ結果が保証される設計になっているか
- 途中失敗後の再実行でも安全か

### 同時実行・競合
- 競合更新（lost update）を防げているか
- 並行実行で順序が入れ替わっても整合性が保てるか
- 集計値・在庫・残高更新が安全か（read-modify-write）


## エラー・異常系

### エラーの扱い
- エラーを握りつぶして成功扱いにしていないか
- 原因や文脈が失われていないか
- 想定内エラーと想定外エラーが区別されているか

### 失敗時の挙動
- 失敗時にどこまで処理が進むかが一貫しているか
- タイムアウト・キャンセル・外部依存失敗時の挙動が明確か
- 再試行の可否と条件が妥当か

### 観測可能性
- 失敗時に必要なログ・メトリクス・トレースが出力されるか
- ログに相関 ID や主要パラメータが含まれているか
- エラー情報の粒度が適切か


## セキュリティ

### 認可・権限
- リソース単位の認可が抜けていないか
- マルチテナント / ユーザー分離が守られているか
- 管理者専用処理の分岐漏れがないか

### 入力検証・境界
- 外部入力を無検証で使用していないか
- バリデーションが境界で行われているか
- 文字列連結でクエリやコマンドを生成していないか

### 情報漏えい
- ログに機密情報（トークン・パスワード・個人情報）が出ていないか
- エラーメッセージに内部構造が露出していないか
- レスポンスに不要な内部フィールドが含まれていないか

### セッション・CSRF・CORS（該当する場合）
- Cookie 属性（Secure / HttpOnly / SameSite）が適切か
- CSRF 対策の前提が崩れていないか
- CORS が過剰に許可されていないか


## テスト

### 重要な振る舞いの担保
- 仕様上重要な振る舞いがテストで観測できているか
- バグ修正に再現テスト・回帰テストが追加されているか
- 破壊的変更に検証手順やテストがあるか

### テスト設計の質
- 実装詳細に依存しすぎていないか
- 時間・順序・乱数・環境依存のテストになっていないか
- 1テストが巨大化していないか

### 粒度と層
- unit / integration / e2e の役割分担が適切か
- モックが多すぎて実挙動を担保できていないか
- 統合テストが重すぎて回しづらくなっていないか


## 将来変更耐性

### 拡張性
- 新要件追加時に分岐の増殖だけで対応する構造になっていないか
- データ構造や API が固定化されすぎていないか
- 差し替えポイントや拡張余地があるか

### 依存と結合
- 特定実装への結合が強すぎないか
- 同じロジックが複数箇所に散在していないか
- 設定値や定数が分散しすぎていないか

### 可読性・意図の明確さ
- 後から読んで意図が追える構造か
- “賢すぎる” 実装で理解コストが上がっていないか
- 例外的な仕様が埋もれていないか

### 計算量・ループ
- 不要なネストループがないか
- データ件数に比例して処理時間が増えすぎないか
- フィルタや集計をアプリ側でやりすぎていないか

### I/O・外部呼び出し
- DB / API 呼び出し回数が不必要に多くないか
- ループ内で I/O を行っていないか
- バッチ化・まとめ取得が可能ではないか

### データ取得・クエリ
- 全件取得してから絞り込んでいないか
- インデックスが前提となるクエリになっているか
- N+1 問題が発生しうる構造になっていないか

### キャッシュ・再計算
- 同じ計算や取得を何度も繰り返していないか
- キャッシュ前提の設計を暗黙に置いていないか
- キャッシュ無効時でも破綻しないか

### メモリ・リソース
- 不要に大きなオブジェクトを保持していないか
- リクエストスコープを超えてデータを保持していないか
- 解放されないリソース（goroutine / listener / timer 等）がないか

### 初期実装としての妥当性
- 現時点の要件に対して過剰最適化していないか
- 逆に、将来確実に詰まる構造を放置していないか



# 対応優先度ガイドライン

## High になりやすい変更
- セキュリティ・認可
- データ破壊・移行
- アーキテクチャ境界違反
- 並行性・排他制御

## Medium になりやすい変更
- テスト不足
- 監視・ログ不足
- エッジケース未考慮

## Low になりやすい変更
- 命名・スタイル
- 軽微な構造改善
