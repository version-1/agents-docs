
### Go のコードを書くときの注意点

#### `any` の扱い

- **`interface{}` ではなく `any` を使う**
  - Go 1.18 以降では `any` は `interface{}` のエイリアスであり、意味が明確
  - 「任意の型を受け取る」という意図がコードから読み取りやすくなる

- **`any` の使用は最小限にする**
  - 可能な限り具体的な型や構造体を定義する
  - 型安全性・可読性を犠牲にしない

- **型パラメータ（Generics）が使える場合は優先する**
  - `any` + 型アサーションよりも、型パラメータの方が安全で明確
  - コンパイル時に型チェックできる設計を選ぶ

- **`any` を使う理由を明確にする**
  - API 境界、汎用ユーティリティ、外部入力など用途を限定する
  - 意図が分かりにくい場合はコメントを添える

- **型アサーションは必ず安全に行う**
  - `v.(T)` の直接使用は避け、`v, ok := v.(T)` を使う
  - 失敗時の挙動を明示的に扱う

- **戻り値に `any` を使うことは特に慎重にする**
  - 呼び出し側の負担が増えやすい
  - 設計が曖昧になっていないか見直す

### `reflect` の扱い

- **`reflect` の使用は最小限にする**
  - 可読性・保守性・安全性が大きく低下しやすい
  - 通常の型・インターフェース・Generics で解決できないか先に検討する

- **`reflect` が許容される場面を限定する**
  - 汎用ライブラリ、フレームワーク、シリアライズ／デシリアライズ処理
  - 設定値のマッピングやメタプログラミング的処理などに限る

- **業務ロジックでは原則使用しない**
  - ドメインロジックに `reflect` を持ち込まない
  - ビジネスルールは静的型で表現する

- **Generics が使える場合は `reflect` より優先する**
  - Go 1.18 以降は型パラメータで多くの用途を代替できる
  - コンパイル時に型安全性を担保できる設計を選ぶ

- **`reflect.Value` / `reflect.Type` の扱いを明確にする**
  - ポインタか値かを常に意識する（`Elem()` の有無）
  - nil チェックや Kind チェックを必ず行う

- **panic を起こさない実装を徹底する**
  - `Kind()` を確認せずに操作しない
  - `IsValid()` / `CanSet()` / `CanInterface()` を適切に使う

- **パフォーマンスへの影響を理解する**
  - `reflect` は通常の処理より遅く、最適化されにくい
  - ホットパスや大量ループ内では使用しない

- **テストを必ず用意する**
  - 型の組み合わせごとにテストケースを用意する
  - 想定外の型が渡された場合の挙動も検証する

- **使用理由をコメントで明示する**
  - なぜ `reflect` が必要なのか
  - 他の方法ではなぜ解決できないのかを残す

- **`reflect` を隠蔽する設計にする**
  - 利用箇所を1か所に集約し、外部に漏らさない
  - 呼び出し側には通常の型安全なAPIを提供する
