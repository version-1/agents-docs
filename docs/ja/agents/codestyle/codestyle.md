
# コーディングスタイルガイドライン

こちらの内容は、一般的なガイドを示すものでプロジェクトで指定がある場合はそちらを優先してください。

- 言語別のガイドラインは languages/*.md を参照してください。
- レイヤー別のガイドラインは layers/*.md を参照してください。

以降の内容は、言語やレイヤーに依存しない、より良いコードを書くための一般的な指針です。

### より良いコードを書くために気をつけること

- **目的を明確にする**
  - 何を解決するコードなのかを先に言語化する
  - 仕様・前提条件・制約をコメントやドキュメントに残す

- **可読性を最優先する**
  - 変数名・関数名は「何を表すか」が一目で分かるようにする
  - 1つの関数・メソッドは1つの責務に絞る
  - ネストを深くしすぎない（早期リターンなどを活用）

- **一貫性を保つ**
  - 命名規則、ディレクトリ構成、コーディングスタイルを揃える
  - 既存コードの流儀に合わせる（独自ルールを増やさない）

- **変更しやすい設計を意識する**
  - 将来変わりそうな部分をハードコードしない
  - 依存関係を最小限にし、疎結合を意識する
  - 条件分岐が増えすぎたら設計の見直しを検討する

- **重複を避ける（DRY）**
  - 同じ処理・ロジックは共通化する
  - ただし、早すぎる共通化は避ける

- **意図をコメントする**
  - 「何をしているか」ではなく「なぜそうしているか」を書く
  - トリッキーな実装や制約がある場合は必ず補足する

- **エラーハンドリングを丁寧にする**
  - 想定される失敗パターンを明示的に扱う
  - エラーメッセージは調査に役立つ情報を含める

- **テストしやすさを考える**
  - ロジックと副作用（I/O、外部APIなど）を分離する
  - テストコードを書ける設計かを常に意識する

- **小さく書いて、こまめに見直す**
  - 最初から完璧を目指さず、動く最小単位で書く
  - リファクタリングを前提にする

- **他人が読むことを前提にする**
  - 数ヶ月後の自分、初見の他人が読めるかを意識する
  - 「説明しなくても伝わるコード」を目指す
