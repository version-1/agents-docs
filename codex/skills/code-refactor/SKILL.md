# Refactor Skill

## 目的
- 既存コードに対して「どうリファクタリングするべきか」の**方針・設計・手順**を提示する。
- **コードの変更案（修正後コード・diff・パッチ・置換テキスト）を出さない**。
- 読みやすさ・保守性・テスト容易性・変更容易性を高めるための判断材料を提供する。

## スコープ
### このスキルが提供するもの
- リファクタリングの **方針**（何を、なぜ、どの順に）
- **分割計画**（小さなステップに分けた実施手順）
- **安全性確保の提案**（テスト、計測、ロールバック、feature flag 等）
- **影響範囲とリスク**の整理
- **完了条件（Definition of Done）** の提示
- **レビュー観点**の提示

### このスキルが提供しないもの（重要）
- 修正後のコード、コードブロック、差分（diff）、パッチ
- 「この行をこう書き換える」などの具体的な置換指示
- PR作成手順、コミット内容の生成、実ファイル編集
- 実装詳細に踏み込みすぎる最適化（必要なら“検討事項”止まり）

## 絶対ルール（ガードレール）
- 出力に **コード（修正版コード / 断片的な置換例 / diff）を含めない**。
- 例示が必要な場合は **擬似コードも禁止**。代わりに「手順」「責務」「関数/モジュール境界」「命名方針」を文章で述べる。
- 参照する際は「対象の場所」を **ファイル名 / 関数名 / 責務単位**で指す（行単位の書き換え指示はしない）。
- “最終形の設計”は描くが、**実装の書き方**（具体的なAPI呼び出し・構文）は書かない。

## 入力
- 対象コード（ファイル、主要関数、関連モジュール）
- 現状の課題（読みづらい、重複、責務過多、テスト困難、変更に弱い等）
- 制約（期限、互換性、パフォーマンス、チーム規約、言語/フレームワーク）
- 望ましい到達点（例：責務分割、依存逆転、テスト導入、循環参照解消）

## 出力フォーマット（テンプレ）
以下の順番で出力する。

1. **要約（1〜3行）**
   - 何が問題で、どういう方向性で直すか

2. **現状の観察（Symptoms）**
   - 例：責務の混在、分岐増殖、暗黙の依存、命名の曖昧さ、重複、境界不明

3. **目標（Goals）**
   - 例：変更容易性、テスト容易性、可読性、拡張性、障害調査容易性

4. **非目標（Non-goals）**
   - 例：挙動変更はしない、性能改善は二次、全面刷新はしない

5. **リファクタ方針（Principles）**
   - 例：単一責務、依存の向き、境界の明確化、ドメインとI/O分離、命名規則

6. **手順（Small Steps Plan）**
   - “安全に戻れる”単位に分割して箇条書き
   - 各ステップに以下を付ける：
     - 目的 / 対象範囲 / 影響範囲 / リスク / 検証方法（テスト・計測）

7. **リスクと対策**
   - 例：隠れた挙動依存、パフォーマンス劣化、互換性破壊、並行開発衝突

8. **完了条件（DoD）**
   - 例：テストが揃う、循環依存が解消、複雑度が下がる、責務が説明できる

9. **レビュー観点（Checklist）**
   - 例：責務は明確か、依存は単方向か、副作用は境界に閉じているか、例外/エラー経路は整理されているか

## 判断基準（よくある観点）
- **責務**：1ユニットが「何をするか」を一言で説明できるか
- **境界**：ドメインとI/O（DB/HTTP/FS/外部API）が混ざっていないか
- **依存**：上位が下位の詳細に引きずられていないか（依存逆転できるか）
- **複雑度**：分岐・ネスト・状態が増えすぎていないか
- **重複**：同じ知識が複数箇所に散っていないか
- **テスト**：主要ロジックが副作用なしで検証できる形か

## 依頼時にあると助かる情報
- “何が辛いか”（変更時の事故、改修が遅い、バグが入りやすい等）
- 仕様の不変/可変ポイント
- 既存テストの有無、カバレッジの状況
- 実行頻度・性能要件（ホットパスかどうか）

## 例（出力イメージ：コード無し）
- 方針：`X` は責務過多なので「入力変換」「ドメイン判断」「副作用」を3層に分離する
- 手順：
  1) 既存の挙動を固定するためにテスト観点を列挙し、現状の期待値を明文化
  2) 副作用境界をインターフェース化し、呼び出し点を集約
  3) 条件分岐のルールをドメイン層に寄せ、データ変換を薄くする
  4) 重複ロジックを“規則”として一箇所に集約し、呼び出し側を単純化
  5) メトリクス（複雑度・依存関係）で改善を確認
