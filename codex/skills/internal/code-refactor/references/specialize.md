# Specialize

## 目的

過剰な抽象化をやめ、具体的なユースケースに合わせた実装へ戻す。

## 使う場面

- interface や generic が 1 実装しか持っていない。
- 抽象化のために呼び出し側が読みにくくなっている。
- 汎用化された API が実際のユースケースより広すぎる。

## 代表的な技法

- 不要な interface を削除する。
- generic を具体型へ戻す。
- 汎用 helper を用途別の関数へ分ける。
- 設定可能な引数を具体的な処理に固定する。

## 注意点

- 近い将来の拡張計画や plugin 境界がある場合は削らない。
- テストや mock のためだけに残っている抽象化は、代替のテスト方法を確認する。
- specialize により重複が復活する場合は、重複のコストと可読性の改善を比較する。

## 報告観点

- 取り除いた抽象化。
- 具体化したユースケース。
- 拡張性をどこまで残したか。
