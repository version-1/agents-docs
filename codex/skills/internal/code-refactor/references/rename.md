# Rename

## 目的

名前を改善し、コードの責務、状態、意図を読み手に伝わるようにする。

## 使う場面

- 変数名、関数名、型名が抽象的すぎる。
- 名前と実際の責務がずれている。
- 呼び出し側から何を渡すべきか、何が返るか分かりにくい。

## 代表的な技法

- 役割を表す変数名へ変更する。
- boolean に肯定形の名前を付ける。
- 関数名を副作用、戻り値、対象ドメインに合わせる。
- 型名や component 名を責務に合わせる。

## 注意点

- プロジェクト内の既存語彙を優先する。
- 外部 API、DB カラム、イベント名、URL など公開契約の rename は影響範囲を確認する。
- 長い名前にすればよいわけではない。文脈から分かる語は重複させない。

## 報告観点

- 変更した名前と改善した読み方。
- 公開契約か内部実装か。
- 参照更新やテストで確認した範囲。
