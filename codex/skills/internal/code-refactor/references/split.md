# Split

## 目的

混ざっている責務を分け、変更理由が異なる処理を別々に扱えるようにする。

## 使う場面

- 1 つの関数、クラス、コンポーネントが複数の関心を持っている。
- テストしたい単位が大きすぎる。
- UI、状態、I/O、ドメイン判断が同じ場所に混ざっている。

## 代表的な技法

- 関数分割。
- コンポーネント分割。
- クラス分割。
- 状態管理と表示処理の分離。
- I/O 境界と純粋ロジックの分離。

## 注意点

- 分割後の名前が責務を説明できない場合は、まだ境界が曖昧。
- 呼び出し階層を深くしすぎない。
- 分割で public API を増やす場合は、本当に外部公開が必要か確認する。

## 報告観点

- どの責務をどこへ分けたか。
- 分割後にテストしやすくなった単位。
- 既存の呼び出し側に対する影響。
