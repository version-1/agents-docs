# Inline

## 目的

過剰な間接参照や一度しか使われない薄い抽象化を戻し、読み手が処理を追いやすくする。

## 使う場面

- 1 行程度の関数や変数が意図を増やさず、読み進める邪魔になっている。
- 抽象化の名前より実処理を直接読んだ方が明確。
- wrapper、adapter、helper が実質的に何も隠していない。

## 代表的な技法

- 一度しか使わない関数を呼び出し元へ戻す。
- 不要な中間変数を削除する。
- 薄い wrapper や pass-through method を削除する。
- 過剰に分割された component や hook を統合する。

## 注意点

- public API、テスト対象、拡張ポイントとして意図的に存在する抽象化は消さない。
- inline により呼び出し元が大きくなりすぎる場合は避ける。
- ログ、計測、認可など横断的な処理を隠している wrapper ではないか確認する。

## 報告観点

- 戻した抽象化と理由。
- 読み手が追う必要のなくなった間接参照。
- 削除していない公開契約。
