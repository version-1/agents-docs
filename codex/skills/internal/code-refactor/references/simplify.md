# Simplify

## 目的

複雑さを減らし、同じ挙動をより短く直接読める形にする。

## 使う場面

- 条件式、ネスト、分岐、ループが読みにくい。
- 同じ判定が複数箇所に散っている。
- 一時変数や中間状態が多く、実際の意図が追いにくい。

## 代表的な技法

- 早期 return / guard clause でネストを減らす。
- 条件式に名前を付ける。
- 否定条件を肯定条件へ寄せる。
- 不要な一時変数や冗長な中間処理を減らす。
- 複雑な boolean 式を意味単位に分ける。

## 注意点

- 短さだけを目的にしない。読み手が条件の意味を追いやすい形を優先する。
- 条件の評価順が副作用や nil / null 安全性に影響する場合は変えない。
- 早期 return を増やす場合は、後続の cleanup や deferred 処理に影響しないか確認する。

## 報告観点

- 減らしたネスト、分岐、重複条件。
- 変えていない挙動や境界条件。
- 実行したテスト、または確認できなかった条件。
