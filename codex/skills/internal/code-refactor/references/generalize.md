# Generalize

## 目的

複数の利用箇所に共通する振る舞いを抽象化し、重複と将来の変更コストを下げる。

## 使う場面

- 同じ責務を持つ実装が複数存在する。
- 利用箇所が増える見込みがあり、差分だけを差し替えたい。
- interface、共通 component、strategy で利用側の分岐を減らせる。

## 代表的な技法

- interface 導入。
- 共通 component 化。
- strategy / policy object の導入。
- generic helper の作成。
- template method や callback の導入。

## 注意点

- 2 箇所未満、または将来増える根拠が弱い場合は急がない。
- 抽象化でドメイン語彙が薄まるなら避ける。
- 型パラメータや callback が増えすぎる場合は、抽象化が合っていない可能性が高い。

## 報告観点

- 共通化した責務。
- 差し替え可能にした変化点。
- 抽象化を導入する根拠と、過剰抽象化ではない理由。
