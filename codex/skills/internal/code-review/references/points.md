
# レビュー観点

数字の若いものほど重要な観点で、レビューの優先順位も高いです。

### 1. 仕様・契約・互換性

- 仕様変更が明示されているか
- 公開 API や既存レスポンスの互換性が壊れていないか
- スキーマ変更が既存データを破壊しないか
- 既存クライアントやバッチの暗黙依存を壊していないか

### 2. データ整合性

- 一意制約、外部キー、チェック制約などの前提が守られているか
- アプリ側バリデーションのみに依存していないか
- Atomic な更新が必要な箇所でトランザクションやロックが適切に使われているか
- 冪等性が必要な操作で二重実行が起きないか
- 競合更新や read-modify-write が安全か

### 3. セキュリティ・権限

- リソース単位の認可が抜けていないか
- マルチテナント / ユーザー分離が守られているか
- 外部入力を無検証で使っていないか
- ログやレスポンスに機密情報が漏れていないか
- CSRF / CORS / Cookie 属性の前提が崩れていないか
- SQL インジェクション、コマンドインジェクション、XXE、オープンリダイレクトなどの脆弱性がないか
- 依存ライブラリの脆弱性がないか

### 4. アーキテクチャ・責務(変更容易性)

- 1つの関数やクラス、パッケージが責務を抱えすぎていないか
- 依存の方向性が一方向に保たれているか
- 高凝集・低結合になっているか
- レイヤー境界を越えていないか
- 関心がが適切に分離されているか
  - プレゼンテーションロジック
  - アプリケーションロジック
  - ビジネスロジック
  - データアクセスロジックなどの技術詳細
- 技術的な関心とドメイン的な関心が分離されているか
- データと振る舞いが適切にカプセル化されているか
- ディレクトリ設計やモジュール設計が一貫して意図が明確であるか
- 単体テストが書きやすい構造になっているか

### 5. テスト・検証

- 仕様上重要な振る舞いがテストで観測できているか
- バグ修正なら再現テストや回帰テストがあるか
- 破壊的変更に検証手順があるか
- テストが実装詳細に依存しすぎていないか
- 層ごとの役割分担が適切か

### 6. エラー処理・異常系・観測可能性

- エラーを握りつぶしていないか
- 想定内 / 想定外エラーが区別されているか
- タイムアウト、キャンセル、外部依存失敗時の挙動が明確か
- ログ、メトリクス、トレースが必要十分か
- 失敗時に原因追跡できる情報が残るか

### 7. 性能

- ループ内 I/O、N+1、全件取得後フィルタなどの性能問題がないか
- リソースリークや不要な保持がないか
- 計算量、I/O 回数、メモリ使用量の増加が許容範囲か

### 8. スタイル・命名

- typo やスタイル違反がないか
- 命名が意味を正しく伝えているか
- コメントが正確で最新か
- 例外的な仕様や特別な理由が明示されているか
