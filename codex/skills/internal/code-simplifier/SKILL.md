# Code Simplifier

## 目的

レビュー指摘や既存差分を起点に、挙動を維持したままコードを簡略化する。
読みやすさ、変更しやすさ、テストしやすさを上げることを優先し、必要以上に大きな設計変更は避ける。

## 入力

次のいずれかを受け取る。

- コードレビュー結果
- `git diff`、`gh pr diff`、コミットハッシュなどの差分
- 対象ファイル、ディレクトリ、関数名
- 期待する制約、検証コマンド、避けたい変更

入力が曖昧で変更範囲を安全に決められない場合だけ、最小限の質問をする。

## 進め方

1. 対象差分と関連コードを読み、現状の挙動、公開契約、テストを把握する。
2. 既存のレビュー指摘があれば、可読性、保守性、効率性、検証不足に分類する。
3. レビュー指摘がない場合は、`code-review` の観点を使って差分中心に問題を洗い出す。
4. サブエージェント起動が明示的に許可されている場合だけ、可読性、保守性、効率性の観点を分けてレビューさせる。
5. 挙動維持を前提に、効果が明確で小さく戻しやすい単位からリファクタリングする。
6. 既存テストまたは適切な検証コマンドを実行する。実行できない場合は理由と残リスクを明記する。
7. 変更したファイル、要約、検証結果、残るリスクを報告する。

## 判断基準

- 仕様、API、データ形式、エラー挙動、ログの意味を不用意に変えない。
- 大きな抽象化より、重複削除、命名改善、早期リターン、関数抽出、スコープ縮小を優先する。
- 条件分岐の置き換えやインターフェース導入は、分岐が増え続ける根拠がある場合に限る。
- コメントは「なぜ」を補う場合だけ追加する。コードから読める説明は追加しない。
- 性能改善は、計算量、I/O 回数、メモリ保持、ホットパスへの影響を説明できる場合に行う。
- テスト追加は、挙動固定や回帰防止の価値が変更コストを上回る場合に行う。

## よく使う技法

- 不要コードの削除
- 重複処理の統合
- 関数、クラス、値オブジェクトの抽出
- 変数名、関数名、型名の改善
- 変数スコープの縮小
- ネスト削減と早期リターン
- マジックナンバーや暗黙ルールの名前付け
- 副作用境界の分離
- データのカプセル化
- ループ内 I/O、N+1、不要な全件処理の解消

## 出力

最終報告は次の順で簡潔に書く。

1. 変更した内容
2. 変更ファイルと主要な行
3. 検証結果
4. 残るリスク、または未実行の検証
