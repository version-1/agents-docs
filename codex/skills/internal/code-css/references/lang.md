
これは CSS 及び CSS Module, Sass などでのコーディングに関するガイドラインです。

# 推奨事項

- レイアウトと appearance（見た目）を分離して考え、子はコンポーネントとして閉じることを推奨します。
- 固定値を使用する場合はスケールを意識して、変数を使用してください。
  - 例: `--spacing-1: 4px;` など。
- レスポンシブを意識して、width や height などの値を固定値で指定しないでください。
  - 親コンポーネントでサイズを限定することを推奨します。
- カラーコードは変数を使用してください。グローバル空間に定義が足りない場合は適宜追加してください。
  - 例: `--color-primary: #ff0000;` など。
- z-index を使用する場合は、スケールを意識して、変数を使用してください。
  - 例: `--z-index-1: 100;` など。

## レイアウトと appearance の分離

CSS を書く前に、その変更が要素の配置・サイズ・流れを変えるものか、色・装飾・質感を変えるものかを分けて判断してください。

- レイアウトには、`display`、`grid`、`flex`、`position`、`inset`、`width`、`height`、`min-*`、`max-*`、`margin`、`padding`、`gap` などを含めます。
- appearance には、`color`、`background`、`border`、`box-shadow`、`opacity`、`font-*`、`text-*`、`filter`、状態ごとの装飾を含めます。
- 同じセレクタ内に両方を書く場合でも、宣言順はレイアウト、appearance、状態・アニメーションの順にまとめ、変更意図を追いやすくしてください。
- 物理的なファイル分割やクラス分割は、再利用性、差分の見通し、コンポーネント境界が明確になる場合に行ってください。小さな単発スタイルまで無理に分ける必要はありません。
- レイアウト責務は親コンポーネントや layout primitive に寄せ、子コンポーネントは自分の内部構造と appearance を中心に閉じてください。
- デザイントークンは appearance の変更理由を隠さない範囲で使い、spacing、color、z-index などの意味が混ざらないようにしてください。

# 非推奨事項

- important を使用しないでください。
- id セレクタを使用しないでください。
- 子要素と親要素が依存するコーディングを避けてください。
  - 例: `.parent .child { ... }` は避けてください。子要素用のクラスを作成してください。
- タグセレクタを使用しないでください。
  - 例: `div { ... }` は避けてください。クラスを作成してください。
- ネガティブマージンを使用しないでください。


## CSS Module を使用する場合

- コンポーネント固有のスタイルはコンポーネントごとに CSS Module を作成して管理してください。
- グローバルなスタイルは、アプリケーションに共通するスタイルを定義するためのグローバル CSS ファイルを作成して管理してください。

## グローバルで管理すべきスタイル

- フォントファミリー、フォントサイズ、行間などのタイポグラフィに関するスタイル。
- カラーコードなどのテーマに関するスタイル。
- レスポンシブデザインに関するスタイル。
- 共通のユーティリティクラス（例: `.hidden`, `.visually-hidden`, `.sr-only` など）。
- ブラウザのデフォルトスタイルをリセットするためのスタイル（例: `normalize.css` など）。
- z-index
