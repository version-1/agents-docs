# Next Developer Review 観点

`code-next-developer-review` では、通常の欠陥検出よりも「次に開発する人が困らないか」を優先して見る。

## 1. 変更箇所の見つけやすさ

- 類似機能を追加するとき、最初に見るべきファイルが自然に分かるか
- エントリポイント、設定、型、テスト、ドキュメントがつながっているか
- 命名やディレクトリ配置が責務を正しく示しているか
- 変更対象が散らばる場合、どこを同時に触るべきか分かる手がかりがあるか

## 2. 追加ケースの足しやすさ

- 1 ケース追加するときに、修正箇所が局所的に収まるか
- 条件分岐やフラグが増えても読み順が破綻しないか
- 既存ケースと新規ケースの違いが型、名前、テストデータで表現されているか
- コピーして少し変えるしかない構造になっていないか

## 3. 前提知識の残し方

- なぜその実装にしたかが、コード、テスト、コメント、ドキュメントのどこかに残っているか
- 業務ルール、外部サービス制約、データ移行制約、運用上の例外が暗黙知になっていないか
- TODO に期限、条件、削除基準、後続作業の入口があるか
- 例外的な処理が通常パスと区別できるか

## 4. 型と契約の読みやすさ

- API、DB、イベント、設定、環境変数、CLI 引数などの契約が読み取れるか
- optional、nullable、zero value、default の意味が曖昧でないか
- 不正な状態を作りにくい型やバリデーションになっているか
- 呼び出し元が内部実装の都合を知りすぎていないか

## 5. テストの案内力

- テスト名とケース名から、守るべき振る舞いが分かるか
- 新しいケースを追加する場所が分かるか
- テストデータが現実的で、仕様理解の助けになっているか
- 実装詳細に寄りすぎて、リファクタリング時の足かせになっていないか

## 6. ローカル検証のしやすさ

- 変更後に何を実行すればよいか分かるか
- 外部依存、環境変数、fixture、seed data の準備が読み取れるか
- 失敗時に原因を追えるエラー、ログ、テスト出力になっているか
- 最小の確認手順と、必要な統合確認の境界が分かるか

## 7. 責務境界の分かりやすさ

- ドメイン判断、入出力変換、永続化、通信、表示、運用都合が混ざっていないか
- 状態とロジックが分離され、変更時に影響範囲を読めるか
- 抽象化が実際の変化点を表しているか
- 層やパッケージをまたいだ依存の向きが直感に反していないか

## 8. レビューでの扱い

- 今回直すべきか、後続課題で十分かを分ける
- 後続課題にする場合、次の開発者が再発見しなくて済む形で残す
- 実装案よりも、迷いを減らす手がかり、境界、契約、検証導線を優先して示す
