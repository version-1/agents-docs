# Quality Review Axes

必要な観点だけを選んで使う。すべてを網羅することより、今回の差分で品質判断に効く観点を深く見る。

## 0. 定量シグナル

定量シグナルは、品質問題の候補を見つけるための補助証拠として使う。数値が悪いこと自体を指摘にせず、変更容易性、テスト容易性、理解負荷、誤用可能性への影響と結びつける。

主なシグナル:

- Cyclomatic Complexity: 独立した実行経路の多さ。分岐追加時の回帰リスクを疑う
- Cognitive Complexity: 人間が制御フローを追う負荷。ネスト、例外、条件の絡みを疑う
- Function / method length: 責務混在、抽出不足、テスト単位の大きさを疑う
- Nesting depth: happy path の読みにくさ、異常系の混在を疑う
- Duplication: 仕様変更時の変更漏れ、分岐条件の不一致を疑う
- Fan-in / fan-out: 変更影響範囲、依存の強さ、抽象化境界の弱さを疑う
- Test coverage: 重要な振る舞いが観測されているかを疑う。coverage 率だけではテスト品質を判断しない
- Lint / static analysis: 未使用コード、複雑度違反、危険 API、型の問題を疑う

代表的な取得元:

- Go: `go test -cover`, `golangci-lint`, `gocyclo`, `gocognit`
- TypeScript / JavaScript: ESLint complexity rules, coverage report, dependency / bundle analysis
- Python: `radon`, coverage report, ruff / pylint
- Ruby: RuboCop metrics cops, SimpleCov

出力では、可能なら次の形に整理する。

| Signal | Result | Interpretation | Action |
|---|---:|---|---|
| Cyclomatic Complexity | 18 | 分岐が集中し、変更時の回帰リスクが高い | 分岐条件を名前付けし、責務ごとに関数抽出を検討 |

未計測の場合は `未計測` と明示し、必要な追加検証を `Quality Gate` か follow-up に回す。

## 1. 責務分離と凝集度

- 1つの関数、型、コンポーネント、パッケージが複数の理由で変更されないか
- ドメイン判断、I/O、永続化、表示、状態管理が混ざっていないか
- 関連するデータと振る舞いが近くにあり、不要な共有状態に依存していないか
- 新しいケースを追加するとき、どこを変更すればよいか局所化されているか

## 2. 結合度と依存方向

- 依存方向がレイヤーやモジュール境界に沿っているか
- 実装詳細が上位層や公開契約へ漏れていないか
- テストや呼び出し側が内部構造に依存しすぎていないか
- 変更の影響範囲が名前、型、インターフェースから予測できるか

## 3. 複雑度と制御フロー

- 条件分岐、ネスト、早期 return、状態遷移が追いやすいか
- 同じ条件判断が複数箇所に散らばっていないか
- 例外ケースが通常ケースを読みにくくしていないか
- 分岐が増えたときに、型、strategy、table-driven data などへ押し戻せる余地があるか

## 4. 契約、型、境界

- API、関数、型、設定、イベントなどの入力と出力が明確か
- nil、空値、ゼロ値、未指定、エラーなどの扱いが契約として読めるか
- 公開契約と内部実装の境界が保たれているか
- 型名やフィールド名が業務上の意味を誤解なく伝えているか

## 5. テスト容易性とテスト品質

- 重要な振る舞いを小さく検証できる構造か
- 外部 I/O、時間、乱数、環境変数、グローバル状態がテストを難しくしていないか
- テスト名、fixture、assertion から保証している仕様が読めるか
- 実装詳細ではなく、守るべき振る舞いを検証しているか
- 回帰時に原因を絞り込みやすい失敗メッセージになっているか

## 6. エラー処理と観測可能性

- エラーの発生源、原因、呼び出し側の取るべき行動が分かるか
- エラーを握りつぶしたり、過度に一般化していないか
- ログ、メトリクス、トレース、戻り値が障害調査に足りるか
- 失敗時の後始末、再試行、冪等性の前提が読み取れるか

## 7. 変更容易性

- 次に追加されそうなケースで、変更箇所が予測しやすいか
- 新しい variant、provider、状態、入力形式を足したときに条件分岐が増殖しないか
- 既存の抽象化が過不足なく、変更に対して安定しているか
- 重複が「意図的な局所性」か「変更漏れを招く重複」か判断できるか

## 8. 読みやすさと説明可能性

- 名前、配置、ファイル分割から役割が分かるか
- コメントは実装の翻訳ではなく、背景や例外理由を説明しているか
- 長い関数や大きいファイルが、意味のある単位に分割できる状態か
- 初見のレビュアーが、変更理由と影響範囲を追えるか

## 9. 運用とローカル検証

- ローカルで再現、実行、検証しやすいか
- 設定、環境依存、外部サービス依存が明示されているか
- 失敗時に安全に戻せるか、影響範囲を確認できるか
- CI、lint、test、dry-run などの品質ゲートに乗っているか
